# Change Log

## [1.2.3] (ethansk fork) ‚Äî "Better Git" category in the Command Palette

- **All commands now carry the `Better Git` category**, so the Command Palette shows them as "Better Git: Next git change", "Better Git: Collapse all worktree / repository sections in Source Control", etc. Typing "Better Git" groups every command of the extension together instead of them being scattered by title. Titles themselves are unchanged, so muscle memory and existing keybindings are unaffected.

## [1.2.2] (ethansk fork) ‚Äî collapse worktree / repository sections on startup

- **New: worktree / repository sections in Source Control now collapse automatically on window open / reload.** When you work with multiple git worktrees or repositories in one window, VS Code's Source Control panel shows each as its own collapsible section and renders them **all expanded** every time the window loads ‚Äî noisy once you have a few worktrees (VS Code has no reliable way to persist the collapsed state, upstream [microsoft/vscode#322318](https://github.com/microsoft/vscode/issues/322318)). The extension now folds them for you shortly after the window loads.
- **How it works (and its honest limits).** There is exactly one built-in command that collapses the repository/worktree section headers ‚Äî `workbench.scm.action.collapseAllRepositories` ‚Äî and it's a *view action*: it only runs when the Source Control view is the active sidebar, so the extension briefly reveals Source Control to run it. Because VS Code discovers repositories asynchronously, the extension **polls until the repositories have populated** (up to ~10s) before collapsing, and briefly watches for late-appearing worktrees during a ~12s startup window. It only auto-runs when there are **2+ repositories** (the multi-worktree case) so single-repo windows are left untouched and unfocused. It's a best-effort workaround for the missing native "remember collapsed state" ‚Äî timing-dependent by nature, and it does focus the Source Control panel when it fires.
- **New setting `better-git-vscode.collapseWorktreesOnStartup`** (boolean, default `true`) toggles the auto-collapse. Turn it off to leave the sections however VS Code renders them.
- **New command `better-git-vscode.collapse-worktrees`** ("Collapse all worktree / repository sections in Source Control") to fold the sections manually any time they've crept back open ‚Äî bind it to a key if you like. Unlike the startup path it has no 2-repo gate.

## [1.2.1] (ethansk fork) ‚Äî fix the v1.2.0 nav regression + real E2E suite

- **Fixed: `next change` stopped advancing / did a 5-line jump on MODIFIED files.** v1.2.0's new-file scroll mode decided "is this a new file?" from the **active tab** but then moved the cursor in **`vscode.window.activeTextEditor`** ‚Äî a different concept (the *focused* editor). In the normal review flow keyboard focus sits in the Source Control panel (SCM single-clicks open files with `preserveFocus`), so those two frequently pointed at **different files**: the cursor stepped invisibly in a stale/hidden editor (looked like a total no-op ‚Äî "not going to next change") or 5-line-stepped a modified file's editor (hijacking hunk navigation). Navigation now decides **and** acts on the same tab-derived editor.
- **The 5-line step is now structurally impossible on modified/deleted/renamed files.** New strict gate: it only fires when the active tab is a **plain single editor** (not a side-by-side diff ‚Äî modified files always open as diffs), showing the **on-disk file** (a deleted file's view is HEAD content under a `git:` uri and is excluded), whose git status is **genuinely whole-file-new** (`UNTRACKED` / `INDEX_ADDED` / `INTENT_TO_ADD`; renames ‚Äî git's delete+add trap ‚Äî fail this check), **and** a visible editor for that exact document exists. Anything else ‚Üí normal hunk navigation, never a silent no-op. Trade-off: a staged-new file opened as its empty-original *diff* now advances to the next file (open the plain file to scroll-step it) ‚Äî modified-file navigation is never compromised.
- **Fixed: stage-then-edit dual state.** A file staged as new (`INDEX_ADDED`) and then edited again has a real hunk diff ‚Äî it now navigates hunk-to-hunk instead of being hijacked into 5-line crawling.
- **Fixed: deleted files dead-ending navigation.** `next change` on a deleted file's view now reliably advances to the next changed file (the active change is resolved from the tab, focus-independently, instead of a possibly-stale focused editor).
- **Stepping is visible now.** The stepped editor is focused (an unfocused editor renders no caret, so steps used to be invisible) and each step reveals the cursor centered.
- **New: real E2E test suite** (`npm test`, `@vscode/test-electron`) that builds a throwaway git repo fixture and pins the behaviour for every file state: modified (hunk nav ‚Äî the regression guard), untracked (¬±5 stepping + edge fall-through), staged-new (plain vs diff), deleted (unstaged + staged), renamed, dual-state, short/empty new files, and the `newFileNavLineJump` setting.

## [1.2.0] (ethansk fork) ‚Äî scroll through newly-added files

- **Next/previous-change now scrolls through a fully-added new file instead of skipping it.** When you're reviewing a brand-new file (untracked, staged-new, or intent-to-add ‚Äî i.e. the whole file is one big new diff), there are no per-hunk changes to jump between, so `next change` / `previous change` used to just fly straight past it to the next file ‚Äî you never got to actually read the new file. Now, on a fully-added file, those shortcuts step the cursor **down/up 5 lines** so you can page through and review the entire file. When you reach the bottom (or top), they fall through to the next (or previous) changed file exactly as before, so your keyboard review flow is uninterrupted.
- **Only affects newly-added files.** Modified files still navigate change-to-change via VS Code's diff navigation as before ‚Äî the new behaviour is gated on the file's git status being a whole-new-file state (`UNTRACKED` / `INDEX_ADDED` / `INTENT_TO_ADD`), detected via the git API.
- **New setting `better-git-vscode.newFileNavLineJump`** (number, default `5`) controls how many lines each step moves. Bump it up to page faster through big new files, or down to `1` for line-by-line.

## [1.1.0] (ethansk fork) ‚Äî last-staged status bar

- **New status bar item showing the last file you staged through the extension.** When you stage-and-advance (`Shift+Alt+.` / `Shift+Alt+,`) the review flies file-to-file so fast it's easy to stage something without noticing. A bottom-left indicator ‚Äî `$(check) Staged: <filename>` ‚Äî now persists for the session showing the **last file staged via this extension**, so you can always see (and undo) what just got staged.
- **Click it to reopen that file's staged diff.** The item is clickable: it reopens the staged (index) diff for the last-staged file via the extension's existing diff machinery (`getFileChanges` + `openChangeEntry`, so it handles newly-added / staged-deleted files too), letting you review and unstage it on the spot. Falls back to opening the file if it's no longer staged.
- **Updates from a single stage chokepoint.** Every stage path the extension performs ‚Äî the editor-title `+` button (`stage-current-file`) and both stage-and-advance commands ‚Äî routes through one helper that records the staged file only *after* `git add` succeeds and *before* the editor advances, so the bar always reflects the file that was actually staged (never a failed/empty stage, never the file you jumped to).
- **New setting `better-git-vscode.showLastStagedInStatusBar`** (boolean, default `true`) toggles the indicator. Flipping it off hides the item live; flipping it back on re-shows the current last-staged file.

## [1.0.4] (ethansk fork) ‚Äî open changes at your position + Ctrl+Shift+G

- **New `better-git-vscode.open-change-at-position` command** ‚Äî the mirror image of the reveal command. From a plain file editor it opens that file's side-by-side **changes (diff)** view scrolled to the **exact cursor + scroll position you were just viewing**, instead of `git.openChange`'s default of jumping to the file's first change. (It captures your cursor/top-line *before* opening the diff, then re-applies them to the diff's right side, clamped to the diff's length ‚Äî same clamping the reveal command uses, since a partially-staged file's content can differ.)
- **`Ctrl+Shift+G` now opens the current file's changes at your position AND the Source Control pane.** It's a composite default binding (built on VS Code's `runCommands`) that runs `better-git-vscode.open-change-at-position` then `workbench.view.scm`. This shadows VS Code's stock "show Source Control" chord (which only did the latter). Bound to `Ctrl+Shift+G` on every platform (not `Cmd` on mac, since `Cmd+Shift+G` is taken there). **Fully overridable** in your own `keybindings.json` like any other default.

## [1.0.3] (ethansk fork) ‚Äî Dvorak mode toggle

- **New `better-git-vscode.dvorakMode` setting (boolean, default false).** Flip it on and the four navigation commands swap from their QWERTY positions to Dvorak-comfortable physical keys ‚Äî no hand-editing `keybindings.json` required:
  - `alt+v` ‚Üí next change (QWERTY default `alt+z`)
  - `alt+w` ‚Üí previous change (QWERTY default `alt+a`)
  - `shift+alt+v` ‚Üí stage-and-next (QWERTY default `shift+alt+.`)
  - `shift+alt+w` ‚Üí stage-and-previous (QWERTY default `shift+alt+,`)
- Implemented purely with VS Code's native `config.better-git-vscode.dvorakMode` when-clauses: each of those four commands now ships two keybinding entries ‚Äî the QWERTY default gated `!config.‚Ä¶dvorakMode` and the Dvorak key gated `config.‚Ä¶dvorakMode` ‚Äî so turning the toggle on disables the old default AND enables the new key in one step (no `-command` minus-entries, no extension restart). This replaces the manual Dvorak override block users (incl. the author) previously kept in their own `keybindings.json`. **User keybindings still take precedence over all defaults.**
- Smart forward/back (`alt+,`/`alt+.`), changed-file nav (`cmd/ctrl+alt+,`/`.`), revert (`alt+q`) and reveal (`alt+r`) are intentionally **left on their defaults** in both modes.
- README updated: the old "manually remap on Dvorak" instructions are replaced with the one-toggle approach (with a note that custom keybindings still work).

## [0.8.4] (ethansk fork)

- New: **`better-git-vscode.reveal-current-file-in-explorer`** ‚Äî **opens the real working file in a normal editor AND reveals/selects it in the Explorer tree**, and crucially it **works from staged diffs**, where VS Code's built-in "Reveal in Explorer" silently does nothing. (Reveal alone only highlights the tree node ‚Äî it also opens the editable on-disk file so you don't have to press Space/Enter after.) The staged side of a diff is a read-only `git:`-scheme virtual document (the index blob) with no node in the `file:`-based Explorer tree, so the built-in reveal has nothing to select ‚Äî a confirmed, still-open upstream bug ([microsoft/vscode#240657](https://github.com/microsoft/vscode/issues/240657)). This command resolves that `git:` uri back to the on-disk `file:` path and reveals *that* via the supported [`revealInExplorer`](https://github.com/microsoft/vscode/issues/94720) command (works from unstaged diffs and plain editors too). Bind it to `cmd+shift+e` with `"when": "isInDiffEditor"`.
- Note: go-to-definition / cmd-click in **staged** diffs is a separate VS Code *by-design* limitation ([microsoft/vscode#34034](https://github.com/microsoft/vscode/issues/34034)) ‚Äî semantic features are disabled on the read-only `git:` index side because results would be inaccurate; they work on the working-tree/unstaged side. Not fixed here (documented in the README).

## [0.8.1] (ethansk fork)

- Fix: "The editor could not be opened because the file was not found" when navigating onto a **staged** file that was newly added or staged for deletion. The staged diff was always built as HEAD ‚Üî index regardless of git status, so the side that legitimately has no content ‚Äî HEAD for a brand-new staged file, the index for a staged deletion ‚Äî pointed `git show` at a blob that doesn't exist, and VS Code's git content provider threw "file not found". The staged diff now picks its sides by git status (newly-added ‚Üí empty original; staged deletion ‚Üí empty modified; modified/renamed/copied ‚Üí HEAD ‚Üî index as before), using git's empty-tree object as the guaranteed-resolvable empty side, exactly as VS Code's own Source Control does.
- Fix: staged **renames and copies** (the "R" rows) now open ‚Äî same "file not found" class of bug. The HEAD side was built at the *new* path, but a rename's HEAD content lives at the *old* path, so `git show HEAD:<newpath>` failed and the diff never opened. It now uses the change's `originalUri` for the HEAD side (correct for rename/copy, identical for everything else).
- Generalised across **all** git states so they're navigable + diffable + badge-able: intent-to-add (treated like a new file), type-changed, and **merge-conflict** files (now included in navigation, opened via the conflict / 3-way merge editor, with the "currently reviewing" badge following them). The `GitStatus` table now documents every state. IGNORED is the only state intentionally skipped.

## [0.8.0] (ethansk fork)

- Fix: next-change navigation jumping to the wrong file (or getting stuck) when a file was partially staged / staged-then-edited ‚Äî such a file appears twice (staged + unstaged) and the position lookup matched by path only, locking onto the wrong copy. Navigation entries are now tagged with their staged/unstaged side, the current side is detected from the active diff, and the matching side is opened.
- New: `shift+alt+z` stages the current file and jumps to the next unstaged file, so you can review-and-stage without clicking the + manually.
- New: `shift+alt+a` ‚Äî the reverse mirror ‚Äî stages the current file and jumps to the *previous* unstaged file, for reviewing bottom-to-top.
- New: a **Stage current file** button (a `+` icon) in the editor title bar ‚Äî top-right, next to the built-in diff/open icons ‚Äî shown whenever a git repo is open. It stages the file you're viewing *without* navigating, so you can stage from the diff with a click. Backed by a stage-only command `better-git-vscode.stage-current-file` (also in the Command Palette). Shares all of stage-and-advance's logic (no-op on a staged file, change-file safety guard, untracked-aware list); at the first file it falls back to the next instead of stranding you.
- Fix: navigation occasionally jumping to the wrong file (not the visually-next one in the Source Control list). The file-ordering compared names with a naive `a < b`, which diverges from VS Code's list view for numbered files (e.g. `item-2` vs `item-10`, `v2` vs `v10`) and some punctuation. Now uses the same numeric, case-insensitive collator VS Code uses (`compareFileNames`), so the order matches the panel exactly. Affected both staged and unstaged navigation.
- New (opt-in, `better-git-vscode.revealStagedInSourceControl`, default off): also highlight/select staged files in the Source Control view while navigating. Works around a VS Code limitation where `scm.autoReveal` can't reveal staged (`git:`-scheme) diffs by briefly making the file the active editor first so auto-reveal selects its row, then opening the staged diff. Tradeoffs: a brief flash, and partially-staged files highlight the unstaged copy. Tracked upstream at microsoft/vscode#320087.
- New: a `‚ñ∂` "currently reviewing" badge on the file you're viewing, shown right on the built-in Source Control rows (and Explorer/tabs) via a `FileDecorationProvider` ‚Äî a supported overlay that needs no patching, no debug port, and triggers no corruption warning. The badge follows you as you navigate. Note: it keys on the file path, so a partially-staged (dual-state) file shows the badge on both its staged and unstaged rows (it marks the file, not the side). The extension now activates `onStartupFinished` so the badge works immediately.
- The "currently reviewing" badge now defaults to a colourful `üîµ` (configurable via `better-git-vscode.currentFileBadge`, set empty to disable) for far better visibility ‚Äî the Source Control panel ignores decoration colours, so an emoji is the most visible option there.
- Navigation now LOOPS: pressing next on the last file wraps to the first (and previous on the first wraps to the last) in a single press, instead of closing the editor and needing an extra press.
- `shift+alt+z` (stage current + advance to next unstaged) now does NOTHING when you're already viewing a *staged* file's diff, instead of jumping to an unstaged file ‚Äî it's only for working through unstaged files.
- The current-file badge default is now a louder `üî¥` (still configurable via `better-git-vscode.currentFileBadge`).
- `shift+alt+z` (stage current + advance to next unstaged): when you stage the LAST unstaged file, it now lands on the PREVIOUS unstaged file instead of closing the editor. It only closes when that was the only unstaged file left.
- Fix: clicking an unstaged file in the Source Control panel and then pressing `shift+alt+z` jumped to the *first* unstaged file instead of the one right after the file you clicked. The command resolved the "current file" from `activeTextEditor`, which doesn't register a Source Control panel click (focus stays on the panel, so the editor reference is stale/empty). It now reads the active diff *tab* ‚Äî the same source navigation uses ‚Äî so it stays in sync with whatever you clicked.
- Fix: an intermittent bug where "previous change" navigation would jump to the *last* change instead of the previous one. The looping logic treated "couldn't locate the current file" (-1) the same as "at the first file" (0) and wrapped to the end. It only triggered when the active diff's staged/unstaged side wasn't readable (so a dual-state file's position was guessed wrong) ‚Äî hence intermittent. Navigation now bails when it can't confidently locate the current file (a re-press works), and only loops at a genuine first/last boundary.
- Fix: untracked (brand-new) files are now included in navigation and stage-and-advance. They were filtered out everywhere (`status !== 7`), so `alt+z` and `shift+alt+z` skipped new files entirely ‚Äî a real gap when reviewing AI-generated changesets, which add lots of new files. New files are now gathered from both the working-tree and untracked groups (deduped) and opened via `git.openChange` like any other change, so you can review, stage, and advance through them too.
- Fix: the "currently reviewing" badge now follows untracked (new) files too. Git opens an untracked file as a plain file editor (`vscode.open`, not a diff ‚Äî there's no original side to compare against), and the badge resolver only recognized diff tabs, so the badge vanished on new files. It now also badges a plain-file tab when that file is an actual change.
- Fix: the "currently reviewing" badge now follows **deleted** files (staged and unstaged), and the resolver is general across change types. Two things were missed: (1) a deletion has no working file, so git opens it via `vscode.open` of the HEAD version ‚Äî a single read-only `git:`-scheme editor, *not* a diff ‚Äî and the resolver only handled diff tabs; (2) the single-editor fallback required a `file:` uri, but a deletion's editor uri is `git:`. The badge resolver now decodes a single editor's uri through the same general helper that handles `git:` paths, and diffs resolve from whichever side exists (modified else original) ‚Äî so modify / add / delete / rename / staged variants are all covered without per-status code.
