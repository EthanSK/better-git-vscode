# Change Log

## [1.2.31] — Source Control tree automation off by default

- **Turned every Better Git Source Control show/hide path off by default.** With `better-git-vscode.experimentalScmTreeStateManagement` left off, activation performs no SCM-state read, discovery timer, view reveal/focus, row selection/walk, or expand/collapse command. The rest of Better Git remains active.
- **Withdrew v1.2.30's automatic exact-state restorer.** Its generic `list.*` traversal became quadratic with many worktrees and could visibly select or operate on Source Control Graph instead of the Changes tree. VS Code still exposes no reliable per-repository/group node API, so shipping that behavior as a broken experiment would not be safe.
- **Kept only a narrow double-opt-in collapse experiment.** Enabling the master switch exposes the manual collapse command; also enabling `collapseWorktreesOnStartup` waits for 2+ repositories to settle, then dispatches `workbench.view.scm` and VS Code's built-in `workbench.scm.action.collapseAllRepositories` exactly once. There are no retries, repository-open listeners, or row-driving commands.
- **Expanded the restart regression to eight repositories/worktrees and three modes.** Across two Extension Development Host launches it proves master-off and enabled-with-collapse-off traces remain empty beyond the old loop interval, while double-opt-in collapse emits exactly the two allowed commands once. Exact-window screenshots and placement checks keep every test window on the MacBook's `Built-in Retina Display` and close only that isolated window.

## [1.2.30] — preserve Source Control expansion state

- **Worktrees, repositories, Changes groups, and Staged Changes groups now reopen exactly as you left them.** Better Git no longer overwrites VS Code's saved tree with its old default-on collapse, and it repairs the current VS Code restart regression that otherwise reopens saved-collapsed repositories and groups.
- `better-git-vscode.collapseWorktreesOnStartup` is now **off by default**. Existing explicit `true` overrides still opt into the legacy collapse-all behavior, and the manual **Collapse all worktree / repository sections in Source Control** command remains available.
- Added a two-launch isolated Extension Development Host regression. It seeds one repository open and one closed plus independently mixed nested group state, exits, relaunches the same workspace/profile, and compares the exact saved expanded-node set after Better Git's restoration.

## [1.2.26] (ethansk fork) — stable caret-owned navigation and restart collapse

- **Removed the visible overshoot-and-return in new files and large diff hunks.** The v1.2.22 workaround forced an off-screen reveal and then returned to the requested line. It looked “jumpy jumpy,” and it mistook a legitimate unchanged viewport (caused by VS Code's sticky/cursor-surrounding context) for a dropped scroll.

- **Repository/worktree headers collapse after window reloads and extension-host restarts by default.** Startup now uses the same plain collapse-all behavior as the manual command. It no longer re-expands the primary through `scm.autoReveal` or opens a preview file as a side effect. VS Code still provides no public API for persisting individual repository header states.
- Expanded the real Extension Development Host suite to 35 tests: sticky-context one-line steps, rapid input, direction reversal, full forward/backward boundaries, wrapped variable-length hunks, wrapped final segments, partially staged phantom-hunk protection, stage-and-Previous landing, renderer-event monotonicity, and final-edge-before-rollover ordering.

## [1.2.22] (ethansk fork) — focus-independent scroll stepping

- **Fixed: new-file and tall-hunk stepping silently failed when the target editor was not focused.** The v1.2.18 workaround called VS Code's global `editorScroll` command, which scrolls only the focused editor widget and ignores the `TextEditor` the extension intended to move. Review commands launched from Source Control, another split, or an unfocused/background VS Code window could therefore pin the caret while leaving the viewport stuck at line 1.
- **Scrolling now uses the public, editor-scoped `TextEditor.revealRange` API and verifies the exact requested viewport on the exact editor object.** When the destination is already visible—the edge case that caused v1.2.18's repeated-next freeze—the extension first reveals a nearby off-screen forcing anchor, waits for that exact viewport, then immediately reveals the real target at the top. Small documents that fully fit the viewport are treated as unscrollable success, and timeout fallbacks verify/retry before pinning the caret to the viewport VS Code actually achieved.

## [1.2.16] (ethansk fork) — pin the "+" stage-and-advance button to a stable rightmost position

- **Fixed: the `+` (stage-current-file-and-advance) button in the editor title bar stopped moving.** Before, it would SHIFT left/right depending on what file/diff/git state you were on, so you couldn't just hover one spot and click — you had to hunt for it. The cause: VS Code's built-in diff up/down "Previous/Next Change" arrows live in the SAME right-aligned editor-title navigation group as the `+`, and they spawn/despawn depending on whether the current tab is a diff (and other state). The `+` had no explicit sort order, so when those arrows appeared or disappeared the `+`'s position slid.
- **The fix pins the `+` as the RIGHTMOST navigation item** by giving it an explicit high sort order (`navigation@100`). The editor-title navigation group is right-aligned and sorts ascending by order, so the highest-order item is anchored to the right edge; the built-in arrows (which VS Code registers at orders 10/11 — confirmed against VS Code source) always render to its LEFT and simply collapse their space when they despawn, WITHOUT moving the `+`. Net effect: the `+` now stays in the exact same on-screen spot across every file type / diff / git state, so you can hover-and-click the same pixel every time.
- The `+`'s visibility condition is unchanged (`gitOpenRepositoryCount != 0`) so its slot is always present in git-repo editors, and the built-in arrows keep spawning/despawning as before — only their position RELATIVE to the `+` is now fixed.

## [1.2.15] (ethansk fork) — fix new-file scroll overshooting to the bottom + make it viewport-derived

- **Fixed: on a brand-new (untracked) file that looked like it was at the TOP, pressing next flew straight past the whole file instead of scrolling down a few lines.** The new-file scroll decided "am I at the edge?" and where to step purely from the **cursor** position, never from what was actually on screen. But the cursor and the visible viewport routinely drift apart in the review flow — when you arrive at a new file by going *backward*, the cursor gets parked at the file's last line while a short file still renders from its top. So you'd see the top, press next, and the extension read "cursor is at the last line → nothing more below → roll to the next file." It looked like it jumped to the bottom / shot past the file.
- **The fix makes new-file scrolling viewport-derived, exactly like the tall-hunk stepper.** Every press now reads the live viewport: next scrolls down a step and only rolls to the next file once the file's **last line is actually on screen**; previous mirrors it (scroll up a step, roll to the previous file only once the **first line is on screen**). The cursor is pinned to the top visible line so it stays glued to what you're reading. This also fixes a long-standing "the first few presses don't seem to scroll" feel on tall new files (the old centered reveal wouldn't move the view until the cursor passed the middle).
- **Fixed a matching edge case in the other direction:** when *previous* wraps around to the **last** file and that file is brand-new, it now lands showing the file's bottom (so the next *previous* steps up through it) instead of skipping the whole file unseen — the same fix that already existed for the ordinary previous-file rollover, now shared by both paths.
- Shorter-than-a-screen files, exactly-screen-height files, and reversing direction mid-scroll all behave correctly and symmetrically now, because the decision is recomputed live from the viewport on every press. Debug logging (the "Better Git VS Code" output channel, setting `better-git-vscode.debugLogging`) now covers the new-file scroll decisions too.

## [1.2.12] (ethansk fork) — fix tall-hunk UP-stepping looping back to the bottom + collapse-worktrees off by default

- **Fixed: stepping UP through a tall hunk (`Alt+,`) looped back to the BOTTOM of the same hunk at the top instead of going to the previous change.** This is the exact mirror of the v1.2.10 down-direction fix, which was only ever applied to the bottom edge. Going up, the code decided "top reached → advance to previous change" purely from the viewport geometry (`remainingAbove <= 0`), with no caret-based signal. When a line or two of viewport render slack left that check just shy of zero, the press re-entered the step path (and could even scroll above the hunk); when the built-in advance did fire it could re-land inside the same hunk and the on-landing reposition re-showed that hunk's bottom — an infinite within-hunk loop that never reached the previous change.
- **The fix makes the up and down branches provably symmetric.** Up now mirrors down exactly: a caret-based "reached the far edge → advance" signal (`caret <= hunk.start`, mirroring `caret >= hunk.end`), a final step that reveals the hunk's **first line at the TOP** and parks the caret there (mirroring the final down-step revealing the last line at the bottom), then the next press advances to the previous change. The symmetry is documented in-code so it can't drift again.
- **Belt-and-braces anti-loop guard:** after any built-in advance, if it re-landed the caret inside the same hunk we were just stepping through, the extension no longer re-reveals that hunk (which was the visible "bounce"). Applies to both directions and all entry points (keyboard `Alt+.`/`Alt+,`, the smart mouse Forward/Back buttons, and the stage buttons — all route through the same navigation).
- **Changed: `better-git-vscode.collapseWorktreesOnStartup` now defaults to OFF (opt-in).** Auto-collapsing the other worktree/repository sections on startup is now off by default; turn the setting on to enable it. The manual `Better Git: Collapse all worktree / repository sections in Source Control` command still works regardless.

## [1.2.11] (ethansk fork) — reliably collapse worktrees on extension-host restart

On a restart the Source Control view re-renders from its saved state at the same instant the extension re-activates, so the single collapse could be undone a beat later. ## [1.2.10] (ethansk fork) — fix tall-hunk stepping getting STUCK near the bottom + debug logging

- **Fixed: stepping DOWN through a tall hunk got permanently stuck a few lines short of the bottom** and never advanced to the next file. On a hunk whose last lines are near end-of-file, the final down-step asked VS Code to put a near-EOF line at the *top* of the viewport — which it can't do (it clamps the scroll, since there aren't a screenful of lines below it). The viewport silently stopped moving, but the "have I reached the bottom?" check still said "no", so every further next-change press was a dead no-op: the last few changed lines never scrolled on screen and it never rolled to the next change/file. (`Alt+,` up-stepping was unaffected and already worked.)
- **The fix guarantees the tail is seen AND that it advances.** The final down-step now reveals the hunk's **last line at the BOTTOM** of the viewport (a downward scroll, which is never EOF-clamped) instead of trying to place an unreachable line at the top — so the tail lines always show — and parks the caret on that last line. The very next press sees the caret at the hunk end and definitively advances to the next change/file. No more no-op loop, no skipped tail.
- **New: a "Better Git VS Code" output channel with optional debug logging** (setting `better-git-vscode.debugLogging`, default off). When enabled, each tall-hunk step logs the direction, viewport height, hunk extent, top/bottom visible lines, caret line, computed target, remaining lines, and the decision taken (stepped / advanced / reached-end). Open it via View → Output → "Better Git VS Code". Off by default so there's zero noise in normal use — flip it on to diagnose any future stepping issue instantly.

- **Fixed: a nav/reveal keypress could throw (do nothing) when the git extension wasn't ready or no repos had loaded** — it now no-ops gracefully like every other git-touching path.
- **Fixed: a late worktree auto-collapse could yank you off the diff you were actively reviewing.** It now skips the re-reveal while a review is in flight.
- **Fixed: rapid navigation while focus was in the Source Control panel could blank/lose your clipboard** — the clipboard-based path lookup is now avoided in the common path (resolved from the active tab instead) and serialized so concurrent presses can't interleave.

## [1.2.8] (ethansk fork) — mouse Forward/Back buttons now navigate **every** review view (new/untracked, deleted, …)

- **Fixed: the smart mouse Forward/Back buttons (`smart-forward` / `smart-back`) only navigated inside modified-file diffs** — on newly-added / untracked ("U") files and deleted files they fell through to plain browser back/forward history and did nothing useful: no new-file 5-line scroll, no advance to the next change. The keyboard `Alt+,` / `Alt+.` worked on all of these, so mouse and keyboard behaved inconsistently.
- **Root cause:** the smart commands decided "am I in a review view?" by checking **only** for a side-by-side diff tab (`TabInputTextDiff`). But VS Code opens a reviewed change in three different shapes: a **diff** (modified/renamed files), a **plain `file:` editor** (whole-new / untracked / staged-new files — they have no original side to diff), and a **plain `git:` editor** (a deleted file's HEAD blob). Only the first matched, so the other two routed to browser navigation.
- **Fix — made general, not a per-case patch.** The mouse buttons are now thin wrappers that engage `next-scm-change` / `previous-scm-change` for **all** the review shapes the keyboard already handles, each recognised by **reusing the extension's existing predicates** (one source of truth, no divergent checks): the diff check for modified/renamed files (so hunk navigation, cross-file rollover, and v1.2.6 tall-hunk staging all work via the mouse for free); `newFileScrollEditor()` for the new-file 5-line scroll; and `isChangeFileUri()` for a deleted file's HEAD view. Because the new-file gate requires a *genuinely-added* file, a **modified file you opened as a plain editor to edit** is deliberately **not** hijacked — its review view is the diff, so the mouse still does normal browser back/forward while you're editing.
- **Direction semantics unchanged.** Forward/Back go exactly the same way they did before — only which views *engage* change-navigation is broadened. Genuine non-review editors (a clean file you're just editing) still get plain browser history navigation, untouched.
- **Note:** the mouse stage-and-advance buttons (bind e.g. F18/F19 to `stage-and-next-changed-file` / `stage-and-previous-changed-file`) already share the same `stageCurrentFileAndAdvance` function as the keyboard `Shift+Alt+.` / `Shift+Alt+,` and the editor-title `+` button, so they inherit every edge case too.

## [1.2.7] (ethansk fork) — the editor-title `+` button now stages **and advances**

- **The `+` button in the editor title bar now stages the current file AND jumps to the next change** — in whatever direction you last navigated. Before, clicking `+` only staged the file and left you sitting on it, so you had to reach for the keyboard to move on. Now the whole review-and-stage flow works with the **mouse alone**: click `+` to stage-and-advance, again and again, sweeping through the changeset without touching the keyboard.
- **It follows your last-navigated direction.** If you last jumped *forward* through changes (`>` / `Alt+.` / next-changed-file), `+` advances forward to the next unstaged file; if you last jumped *backward* (`<` / `Alt+,` / previous-changed-file), `+` advances backward. A fresh session with no navigation yet defaults to advancing **forward** (top-to-bottom, the common case).
- **Identical to the keyboard shortcut.** The button runs the **exact same** underlying function as `Shift+Alt+.` / `Shift+Alt+,` (stage-and-next / stage-and-previous) — same staging, same advance target, same cross-file rollover, same "last staged" status-bar record. Both the keyboard commands and the button share one `stageCurrentFileAndAdvance(direction)` implementation, so they can never drift apart.
- **Safe on non-diff editors.** If there's nothing stageable under the cursor, the button quietly does nothing (never an error) — its change-detection guard is the same one the keyboard shortcuts use.
- The plain, no-advance `Stage current file` command is still registered (bind it to a key if you preferred the old button behaviour); it's just no longer what the `+` button runs.

## [1.2.6] (ethansk fork) — step through TALL hunks in stages (stage-scrolling)

- **New: a hunk taller than your screen now steps through in stages using the same next/previous-change keys.** Before, one press of next-change landed you at the *top* of a tall hunk and the rest ran off the bottom — so you had to take your hands off the keyboard, scroll manually, then press next again. Now: next-change lands at the top of a tall hunk; pressing next **again scrolls down about a screenful within that same hunk**; repeat until the bottom is on screen; the *next* press then advances to the next hunk. `previous-change` mirrors it — it steps **up** through a tall hunk in stages, and once the top is visible the next press goes to the previous hunk. Reversing direction mid-scroll feels natural (pressing previous while stepping down a hunk steps back up — it never teleports).
- **Only tall hunks stage.** A hunk that already fits on screen behaves **exactly as before** — one press goes straight to the next/previous hunk. Staging engages only when the hunk is taller than your **visible editor** (measured live from the viewport, so it adapts to your window size). New-file line-scrolling, modified-file hunk nav, cross-file rollover, the smart mouse buttons, and Dvorak/QWERTY key gating are all untouched.
- **Reading context carries across each step.** Each step scrolls ~one viewport *minus a few lines of overlap*, so you never lose your place with a blind full-page jump. If only a sliver of the hunk remains past the edge, it skips the pointless tiny step and just advances.
- **How it knows a hunk's height.** VS Code's built-in change navigation only exposes hunk *starts*, so the extension parses the file's unified diff from the git API to learn each hunk's full extent (the modified/right-side line span). Everything is best-effort — if the diff can't be read, it silently falls back to plain hunk-to-hunk navigation, never a dead keypress.
- **New settings** (all under **Settings → Better Git VS Code**):
  - `better-git-vscode.hunkStagingEnabled` (boolean, default **true**) — master toggle for the whole feature.
  - `better-git-vscode.hunkStagingThreshold` (number, default **0** = *auto*, use the live viewport height) — minimum hunk height in lines before staging engages; set a positive number to override with a fixed value.
  - `better-git-vscode.hunkStagingLineStep` (number, default **0** = *auto*, one viewport minus the overlap) — lines scrolled per within-hunk step; set a positive number for a fixed step.
  - `better-git-vscode.hunkStagingOverlap` (number, default **4**) — lines of overlap kept between steps so reading context carries across.

## [1.2.5] (ethansk fork) — fix reversed/broken `>` `<` change navigation on QWERTY

- **Fixed: on QWERTY keyboards the headline `>` / `<` keys (`Alt+.` / `Alt+,`) navigated changes BACKWARDS (and did editor history navigation outside a diff).** Root cause: those keys were bound to `smart-forward` / `smart-back` — the **mouse-button** commands whose in-diff direction was *intentionally reversed* for mouse thumb-buttons in v0.8.3 (commit `6043d05`). The keyboard keys silently inherited that mouse flip, so pressing `>` went to the *previous* change and `<` to the *next*, and outside a diff the keys fell back to `navigateForward`/`navigateBack` instead of change navigation. **Dvorak keyboards were never affected** — their physical `>`/`<` keys type `v`/`w`, which were bound straight to the real `next-scm-change`/`previous-scm-change` commands. That correct Dvorak behaviour is the source of truth and is now reverse-mapped onto QWERTY.
- **QWERTY `Alt+.` / `Alt+,` now run `next-scm-change` / `previous-scm-change` directly** — the exact same canonical commands (hunk nav, new-file scrolling, cross-file rollover) that Dvorak mode's `Alt+V` / `Alt+W` run. One behaviour, defined once; each layout is now purely a *character remap of the same physical keys* (Dvorak mode = thin key wrapper, never a behaviour fork). Dvorak bindings and behaviour are completely unchanged.
- **Dropped the legacy QWERTY `Alt+Z` / `Alt+A` defaults** for next/previous change. `Alt+Z` was shadowing VS Code's built-in **Toggle Word Wrap** (that default now works again); the `>`/`<` keys are the advertised home for change navigation. Rebind in `keybindings.json` if you miss them.
- **`smart-forward` / `smart-back` are now mouse-only commands with no default QWERTY keyboard binding** (their reversed direction is a thumb-button ergonomic, not a keyboard one). In Dvorak mode they keep `Alt+.` / `Alt+,` — on Dvorak those characters sit on different physical keys, so nothing collides and existing Dvorak setups are preserved exactly. Bind your mouse Forward/Back buttons to them (e.g. Karabiner → F13/F17).

## [1.2.4] (ethansk fork) — keep the primary/main repo expanded when collapsing worktrees

- **The auto-collapse now leaves your PRIMARY / main repository expanded and folds only the OTHER worktrees.** Previously (v1.2.2) the startup collapse folded *every* repository/worktree section — including the main working copy at the top, the one you're actually working in. Now the top/primary repo stays expanded and only the additional linked worktrees below it collapse. The manual **Collapse all worktree / repository sections** command does the same (keeps the primary open).
- **How it detects the primary.** It's the repository whose root matches your first workspace folder (`vscode.workspace.workspaceFolders[0]`) — your main working copy. Linked worktrees live outside the workspace folder (e.g. `~/Documents/claude-worktrees/…`) so they don't match. If that can't be resolved it falls back to the first repository the git extension discovered.
- **How it keeps the primary expanded (and the honest limits).** VS Code has **no** command to collapse or expand a *single* repository — the only built-ins (`collapseAllRepositories` / `expandAllRepositories`) act on all of them at once. So the extension collapses everything, then re-expands *just* the primary by nudging VS Code's own `scm.autoReveal`: it opens one of the primary repo's changed files in a preview tab (preview + `preserveFocus`, so keyboard focus stays on Source Control), which makes the Source Control view `expandTo` that file — re-opening the primary's header while every other worktree stays folded. Consequences, stated plainly: it relies on **`scm.autoReveal`** being on (VS Code default true) — if you've turned it off, the primary collapses with the rest; it opens the primary repo's first change in a **preview tab** (a reasonable landing spot for a diff-review tool, but it is an extra tab); and if the primary has **no changes**, there's nothing to reveal toward, so it stays collapsed. All best-effort and wrapped in try/catch — it never breaks startup.

## [1.2.2] (ethansk fork) — collapse worktree / repository sections on startup

- **New: worktree / repository sections in Source Control now collapse automatically on window open / reload.** When you work with multiple git worktrees or repositories in one window, VS Code's Source Control panel shows each as its own collapsible section and renders them **all expanded** every time the window loads — noisy once you have a few worktrees (VS Code has no reliable way to persist the collapsed state, upstream [microsoft/vscode#322318](https://github.com/microsoft/vscode/issues/322318)). The extension now folds them for you shortly after the window loads.
- **How it works (and its honest limits).** There is exactly one built-in command that collapses the repository/worktree section headers — `workbench.scm.action.collapseAllRepositories` — and it's a *view action*: it only runs when the Source Control view is the active sidebar, so the extension briefly reveals Source Control to run it. Because VS Code discovers repositories asynchronously, the extension **polls until the repositories have populated** (up to ~10s) before collapsing, and briefly watches for late-appearing worktrees during a ~12s startup window. It only auto-runs when there are **2+ repositories** (the multi-worktree case) so single-repo windows are left untouched and unfocused. It's a best-effort workaround for the missing native "remember collapsed state" — timing-dependent by nature, and it does focus the Source Control panel when it fires.
- **New setting `better-git-vscode.collapseWorktreesOnStartup`** (boolean, default `true`) toggles the auto-collapse. Turn it off to leave the sections however VS Code renders them.
- **New command `better-git-vscode.collapse-worktrees`** ("Collapse all worktree / repository sections in Source Control") to fold the sections manually any time they've crept back open — bind it to a key if you like. Unlike the startup path it has no 2-repo gate.

## [1.2.1] (ethansk fork) — fix the v1.2.0 nav regression + real E2E suite

- **Fixed: `next change` stopped advancing / did a 5-line jump on MODIFIED files.** v1.2.0's new-file scroll mode decided "is this a new file?" from the **active tab** but then moved the cursor in **`vscode.window.activeTextEditor`** — a different concept (the *focused* editor). In the normal review flow keyboard focus sits in the Source Control panel (SCM single-clicks open files with `preserveFocus`), so those two frequently pointed at **different files**: the cursor stepped invisibly in a stale/hidden editor (looked like a total no-op — "not going to next change") or 5-line-stepped a modified file's editor (hijacking hunk navigation). Navigation now decides **and** acts on the same tab-derived editor.
- **The 5-line step is now structurally impossible on modified/deleted/renamed files.** New strict gate: it only fires when the active tab is a **plain single editor** (not a side-by-side diff — modified files always open as diffs), showing the **on-disk file** (a deleted file's view is HEAD content under a `git:` uri and is excluded), whose git status is **genuinely whole-file-new** (`UNTRACKED` / `INDEX_ADDED` / `INTENT_TO_ADD`; renames — git's delete+add trap — fail this check), **and** a visible editor for that exact document exists. Anything else → normal hunk navigation, never a silent no-op. Trade-off: a staged-new file opened as its empty-original *diff* now advances to the next file (open the plain file to scroll-step it) — modified-file navigation is never compromised.
- **Fixed: stage-then-edit dual state.** A file staged as new (`INDEX_ADDED`) and then edited again has a real hunk diff — it now navigates hunk-to-hunk instead of being hijacked into 5-line crawling.
- **Fixed: deleted files dead-ending navigation.** `next change` on a deleted file's view now reliably advances to the next changed file (the active change is resolved from the tab, focus-independently, instead of a possibly-stale focused editor).
- **Stepping is visible now.** The stepped editor is focused (an unfocused editor renders no caret, so steps used to be invisible) and each step reveals the cursor centered.
- **New: real E2E test suite** (`npm test`, `@vscode/test-electron`) that builds a throwaway git repo fixture and pins the behaviour for every file state: modified (hunk nav — the regression guard), untracked (±5 stepping + edge fall-through), staged-new (plain vs diff), deleted (unstaged + staged), renamed, dual-state, short/empty new files, and the `newFileNavLineJump` setting.

## [1.2.0] (ethansk fork) — scroll through newly-added files

- **Next/previous-change now scrolls through a fully-added new file instead of skipping it.** When you're reviewing a brand-new file (untracked, staged-new, or intent-to-add — i.e. the whole file is one big new diff), there are no per-hunk changes to jump between, so `next change` / `previous change` used to just fly straight past it to the next file — you never got to actually read the new file. Now, on a fully-added file, those shortcuts step the cursor **down/up 5 lines** so you can page through and review the entire file. When you reach the bottom (or top), they fall through to the next (or previous) changed file exactly as before, so your keyboard review flow is uninterrupted.

- **New setting `better-git-vscode.newFileNavLineJump`** (number, default `5`) controls how many lines each step moves. Bump it up to page faster through big new files, or down to `1` for line-by-line.

## [0.8.1] (ethansk fork)

- Fix: "The editor could not be opened because the file was not found" when navigating onto a **staged** file that was newly added or staged for deletion. - Fix: the "currently reviewing" badge now follows untracked (new) files too. Git opens an untracked file as a plain file editor (`vscode.open`, not a diff — there's no original side to compare against), and the badge resolver only recognized diff tabs, so the badge vanished on new files. It now also badges a plain-file tab when that file is an actual change.
- Fix: the "currently reviewing" badge now follows **deleted** files (staged and unstaged), and the resolver is general across change types. Two things were missed: (1) a deletion has no working file, so git opens it via `vscode.open` of the HEAD version — a single read-only `git:`-scheme editor, *not* a diff — and the resolver only handled diff tabs; (2) the single-editor fallback required a `file:` uri, but a deletion's editor uri is `git:`. The badge resolver now decodes a single editor's uri through the same general helper that handles `git:` paths, and diffs resolve from whichever side exists (modified else original) — so modify / add / delete / rename / staged variants are all covered without per-status code.
